# 技术能力证据化改造设计

日期：2026-07-15

## 1. 目标

技术能力板块面向招聘者和技术面试官，核心不是证明梁伟业知道多少技术名词，而是让访问者快速判断他能否把 Agent 功能做成可运行、可验证、可恢复、可交付的工程链路。

本次保留暖白、黑色、编辑式排版和单项展开交互，不重做视觉语言。改造集中在信息架构、文案可信度和项目证据联动。

## 2. 信息架构

现有五项能力合并为四条交付链路：

1. Agent 工作流与路由
2. RAG 检索与评估
3. 受控工具执行与异常恢复
4. Agent 应用全栈交付

“工具与约束”和“可靠性与安全”合并。当前项目证据主要覆盖工具约束、输出校验和失败兜底，不单独使用“安全”这一范围更大的能力声明。

板块说明改为：

> 我擅长的不只是接入模型，而是把 Agent 链路做到可编排、可评估、可恢复、可交付。

## 3. 折叠行

折叠态优先展示可完成的工程动作，不以工具清单充当能力描述。技术栈仅作为次级信息出现。

### 3.1 Agent 工作流与路由

- 能力句：用显式状态图组织意图识别、条件路由与专业 Agent 调度
- 技术栈：LangGraph · LangChain · SSE

### 3.2 RAG 检索与评估

- 能力句：从混合召回、RRF 融合到事实核验与失败补充检索
- 技术栈：Milvus · BM25 · FastAPI

### 3.3 受控工具执行与异常恢复

- 能力句：让模型写入可校验、可撤销，执行失败时可恢复
- 技术栈：Tool Calling · JSON Schema · Result Validation

### 3.4 Agent 应用全栈交付

- 能力句：从后端工作流、状态存储到流式接口与前端反馈
- 技术栈：FastAPI · React · TypeScript · Docker

## 4. 展开内容

三栏标签统一使用“工程问题 / 关键实现 / 项目证据”，每栏回答一个问题，避免 Approach、Methods、Proof 与中文内容关系含糊。

### 4.1 Agent 工作流与路由

- 工程问题：把意图分类、业务路由、专业 Agent 调用和回复生成拆成显式节点，避免一条 Prompt 承担整条流程。
- 关键实现：StateGraph 条件路由；分类结果规范化；Checkpoint 会话恢复。
- 项目证据：多智能体客服｜意图识别由约 70% 提升至约 95%，完成“分类—路由—专业 Agent—SSE 回传”链路。

### 4.2 RAG 检索与评估

- 工程问题：完成从数据入库、混合召回、结果融合到检索评估的完整链路，并为低质量召回设计补充搜索和重试路径。
- 关键实现：Dense + Sparse 混合召回；RRF 结果融合；查询改写与并行子查询；检索失败补充与事实核验。
- 项目证据：医疗 RAG｜召回率 91.3%，精确率 92.1%。

### 4.3 受控工具执行与异常恢复

- 工程问题：把简历读写从自由文本生成改为受约束的工具调用，确保每次写入可校验、可追踪、可撤销。
- 关键实现：字段级输出校验；工具权限与参数约束；写入前生成快照；失败时撤销恢复。
- 项目证据：CareerForge-AI｜实现简历读取、局部更新、写前快照、撤销恢复与 PDF 导出。

### 4.4 Agent 应用全栈交付

- 工程问题：从 Agent 工作流、状态存储和流式接口，一直交付到前端反馈、异常展示与部署运行。
- 关键实现：FastAPI 接口与状态管理；SSE 流式结果回传；前后端数据契约；Docker 化部署。
- 项目证据：多智能体客服 / CareerForge-AI｜完成核心 Agent 链路、流式接口与产品界面联调。

## 5. 证据与术语边界

- 每条证据必须包含项目名称、个人完成的工程动作和可验证结果。
- 有真实测量时使用数字；没有测量时展示已交付链路，不制造性能指标。
- `MCP`、`Skills`、`GraphRAG`、`Neo4j`、`Prompt Injection Defense` 不在该板块作为主要能力标签展示，除非后续补充能够在项目中验证的实现证据。
- 删除“毫秒级首字反馈”；当前没有延迟测量条件，改为可验证的 SSE 流式回传事实。

## 6. 交互

- 默认展开第一项“Agent 工作流与路由”，与首屏的 Agent 应用开发定位一致。
- 仍然一次只展开一项，并保留现有加减号、键盘操作和展开动画。
- 每项项目证据提供“查看项目”操作；点击后滚动到项目板块并展开对应项目。
- “Agent 应用全栈交付”涉及两个项目时，提供两个明确的项目入口，不使用含糊的单一跳转。

## 7. 响应式与可访问性

- 桌面端保持现有三栏展开布局；移动端按“工程问题—关键实现—项目证据”自然纵向排列。
- 项目入口、展开按钮必须可由键盘操作，并具有明确可见的焦点状态和可读名称。
- 展开或跳转不得产生横向滚动，也不得为了维持一屏高度裁切内容。

## 8. 验收标准

- 页面仅显示四条能力，顺序与本设计一致。
- 折叠态首先说明能完成的工程动作，技术栈为次级信息。
- 展开态三栏标题与内容语义一致。
- 所有能力证据可以追溯到现有项目内容，没有无法验证的技术名词或性能指标。
- 默认展开第一项，一次只展开一项。
- “查看项目”能够滚动到正确项目并展开该项目。
- 桌面和移动布局均完整可读；键盘展开、项目跳转和焦点状态可用。
